# "Multi-Agent-Debate-System" (-> danwa)

## Frage (2026-04-22):

Ich suche nach einem Agenten-Chat, der für konkretes Problemlösen konzipiert ist (also nicht "Moltbook"-Spielereien oder Entertainment, sondern ein ernsthaftes Werkzeug für die produktive Verwendung).

Szenario:
- Als Input Dokumente/Briefe oder eine ausführliche Sachverhaltsschilderung, bspw. eine rechtliche Thematik;
- 3 KIs (zufällige Zahl) erörtern den Sachverhalt untereinander, z.B. KI 1 entwirft eine gerichtliche Strategie, KI 2 zerlegt die Argumentation und gib an KI 1 zurück, KI 3 optimiert die Strategie…; oder feste Rollenaufteilung wie bei Grok (Harper, Lucas, Benjamin); Opportunismus- und Echokammer-Problematik müsste gelöst werden können.
- Als Endresultat kommt im Ideal etwas heraus, was stichhaltiger ist als die Tagesform-Produkte von Durchschnittsanwälten.

Im konkreten Fall wäre bspw. wichtig, dass
- Reasoning/Diskussion klar nachvollziehbar ist und protokolliert wird;
- zumindest ein Minimal-Memory-System existiert;
- PDF- und Libreoffice-Dateien gelesen werden können;
- grundlegende agentische Tools wie Internet-Suche bereits implementiert sind;
- Dokumente geschrieben werden können, mindestens als TXT;
- keine Bindung an LLM-Provider wie OpenAI, also Unterstützung für lokale LLMs via LM Studio, Openrouter etc.

Anscheinend gibt es kein UI für GNU/Linux, oder? Ich muß dann vermutlich etwas selbr basteln mit Chainlit oder Chainlit. Was würdest du empfehlen, und welche Schritte zur Projektplanung empfiehlst du?

## Antwort (Qwen3.6-Plus):

Es gibt aktuell kein fertiges All-in-One-UI, das Ihr Szenario (multi-agenten-Debatte, dokumentenbasiert, provider-agnostisch, transparente Reasoning-Traces, Linux-native) vollständig abdeckt. Der Markt bewegt sich stark in Richtung Frameworks, die Sie selbst orchestrieren müssen. Das ist kein Nachteil: Für ernsthafte, auditierbare Workflows ist eine maßgeschneiderte Architektur ohnehin zwingend.
Nachfolgend eine praxisorientierte Empfehlung, wie Sie das System aufsetzen, welche Komponenten sich bewähren und wie Sie Echo-Kammer/Opportunismus strukturell unterbinden.


### Echo-Kammer & Opportunismus strukturell unterbinden

Multi-Agent-Systeme neigen zu Konsens-Bias. Für juristische/analytische Workflows müssen Sie Diversität + Validierung hart verdrahten:

1. Adversarial Role Design: KI 2 = „Devil’s Advocate / Gegenpartei“, KI 3 = „Prozessuale Plausibilitätsprüfung“. Prompts explizit auf Widerspruch trainieren.
2. Temperatur-Diversität: Rollen mit unterschiedlichen temperature/top_p befeuern (z.B. Strategie: 0.4, Kritik: 0.7, Optimierung: 0.3).
3. Consensus/Voting Step: Nach N Runden zwingt ein Moderator-Agent zur strukturierten Abstimmung. Bei Abweichung >X% wird eine 4. KI als „Unparteiischer Gutachter“ eingeschaltet.
4. External Validation Layer: Vor Finalisierung läuft ein Tool-Call gegen Faktenbank/Gesetzestext-Suche. Widersprüche führen zu revision_loop.
5. Prompt-Isolation: Keine gemeinsamen System-Prompts. Jede KI erhält kontextbegrenzte Instructions. Cross-Contamination wird so minimiert.

###  Ist Docker zwingend notwendig?

Nein. Docker ist ein Werkzeug, keine Voraussetzung. Auf GNU/Linux ist es oft sogar kontraproduktiv, wenn Sie maximale Transparenz, direkte Dateisystem-Zugriffe und volle Kontrolle über Logs/Prozesse benötigen.

Warum Docker oft empfohlen wird:

- Dependency-Isolation: Ja, aber uv/pipx + Python Venvs lösen das sauberer
- Einheitliche Deployment-Pipeline: Nur relevant, wenn Sie später an mehrere Server ausliefern
- Einfache Netzwerk-Orchestrierung (UI ↔ LLM ↔ DB ↔ Search): Ja, aber systemd + nginx/caddy + lokale Sockets sind auf Linux ebenso stabil
- Sandbox/Security: Teilweise, aber Container teilen den Kernel; für sensible Daten ist native Hardening oft transparenter

### Native Linux-Alternative (empfohlen für Ihre Anforderungen):

#### 1. Python-Env & Abhängigkeiten isoliert
uv venv .venv --python 3.11
source .venv/bin/activate
uv pip install chainlit crewai langgraph langfuse unstructured pdfplumber odfpy

#### 2. Services lokal starten (Beispiel)
- LLM-Router: `lm-studio` oder `ollama serve` (Systemd-Service)
- SearXNG: `docker` ist hier am einfachsten, alternativ native Installation via `pipx install searxng`
- LangFuse: `langfuse` CLI oder Docker, aber auch als `npm`-basiertes Self-Hosting möglich
- Chainlit: `chainlit run app.py --port 7860` (als User-Service unter systemd)

#### 3. Prozess-Management
systemd --user enable chainlit
systemd --user start chainlit
journalctl --user -u chainlit -f  # Live-Logs, direkt durchsuchbar

### Vorteile nativer Aufstellung:

- Keine Abstraktionsebene: Logs, Dateien, Netzwerk-Stacks sind direkt einsehbar
- Einfache Backup/Restore-Strategie (symlinks auf /srv/agent-data, rsync-Jobs)
- Vollständige Kontrolle über Prompt-Versionen, Trace-Exports, LLM-Routing
- Besser kompatibel mit Audit-Anforderungen (kein „black box“-Container-Layer)

### Konkrete Empfehlung für Ihr Projekt

1. Framework: LangGraph (MIT) + manuelle Rollen-Definition. Gibt Ihnen expliziten State, zwingende Validation-Steps, und lückenlose Trace-Exporte. Alternativ smolagents (Apache 2.0), wenn Sie weniger Boilerplate wollen.

2. Setup: Native Linux-Installation mit uv + systemd. Nur SearXNG & LangFuse bei Bedarf in Containern, da deren Dependency-Trees komplex sind.

3. Audit-Design: Jeder Agent-Schritt schreibt JSON-Trace in /var/log/agent-workflow/
LangFuse als optionales Web-Frontend, aber primär lokale .jsonl-Logs + jq/ripgrep für Analyse
Prompt-Versionierung via Git (prompts/strategy_v3.md, etc.)

### Leitprinzipien

1. Auditierbarkeit vor Automatisierung: Jeder Agenten-Schritt, Prompt, Tool-Call und Konsens-Score wird strukturiert protokolliert. Keine Black-Box-Loops.

2. Bias-Strukturen statt Bias-Versprechen: KI ist nicht neutral, aber der Workflow kann Neutralität erzwingen (adversarial roles, Temperatur-Diversität, externe Validierung, konsensbasierte Abbruchlogik).

3. Provider-Agnostik & Native Linux: litellm für einheitliches Routing, uv für Dependency-Management, systemd --user für Prozesse, Chainlit für UI. Zero Docker.

4. Domain-Unabhängigkeit: Prompt-Templates sind parametrisierbar. Recht, Philosophie, Wissenschaft → nur andere Rollen-Beschreibungen & Validierungstools.

### Architektur & Tech-Stack

- LLM-Routing: litellm (MIT) - Unified API für 100+ Provider (LM Studio, OpenRouter, OpenAI, Anthropic, Qwen, DeepSeek, etc.). Handelt Fallbacks, Rate Limits, API-Keys transparent.

- Agent-Orchestrierung: CrewAI (MIT Core) + Custom DebateOrchestrator - CrewAI liefert Rollen/Tools, der Orchestrator steuert State, Runden, Voting & Traces explizit.

- UI: Chainlit - Async-Streaming, Step-Visualisierung, File-Upload, Session-Management, Linux-native.

- Doc-Parsing: unstructured + pdfplumber + odfpy - Industriestandard, ODF-Support, fallback-freundlich.

- Search/Tools: duckduckgo-search + optional SearXNG - Datenschutzkonform, keine API-Keys, zitierfähig.

- Memory/Logging: SQLite (Metadata) + .jsonl (Traces) + in-memory context - Minimal, exportierbar, audit-sicher. LangFuse optional später.

- Output: .txt / .md mit strukturierter Gliederung - Menschlich lesbar, maschinell parsebar, versionierbar.

### Bias-Mitigation (Kernlogik)

- Adversarial Prompting: Kritiker erhält explizit: Du musst mindestens 3 fundamentale Annahmen der Strategie angreifen. Keine Zustimmung. Nur Lücken, Widersprüche, fehlende Präzedenzfälle.
- Konsens-Metrik: Moderator bewertet Übereinstimmung auf Skala 0–1. Bei < 0.7 → neue Runde. Bei > 0.9 → Finalisierung.
- Externe Validierung: Vor Finalisierung prüft ein Tool-Call, ob zentrale Behauptungen in aktuellen Quellen/Texten gestützt sind. Fehlt Stütze → revision_flag=True.

## PDF/ODF-Parser

### 🐧 System-Abhängigkeiten (Linux)

pdfplumber und odfpy sind Python-Pakete, benötigen aber unter Linux teilweise native Bibliotheken:

- Ubuntu/Debian: sudo apt install poppler-utils libxml2-dev
- Arch Linux: sudo pacman -S poppler libxml2
- Fedora/RHEL: sudo dnf install poppler-utils libxml2-devel

(Optional, aber empfohlen für stabile PDF-Layout-Extraktion)

### Workflow-Integration

#### Upload
- Chainlit speichert Files temporär, übergibt Pfade an Parser
- Keine Cloud-Uploads, alles lokal

#### Parsing
- pdfplumber/odfpy/pypdf extrahieren Text + Metadaten
- Fallback zu Plain-Text bei Fehlern

#### Kontext-Injektion
- Dokumente werden strukturiert in context eingebettet
- Metadaten (Autor, Seiten, Wortzahl) erhalten Kontext-Boundaries

#### Overflow-Schutz
- MAX_CONTEXT_CHARS = 25000 verhindert LLM-Absturz
- Explizite Kennzeichnung, wenn gekürzt wurde

#### Trace
- Parser-Ergebnisse fließen in debate_engine.run() → JSONL-Log
- Vollständig nachvollziehbar, welche Textbasis genutzt wurde

### Test-Flow

1. uv run chainlit run src/ui/chainlit_app.py --port 7860
2. UI öffnen → Settings anpassen
3. Klicke 📎, lade eine .pdf, .odt oder .docx hoch

5. Beobachte: 📄 Verarbeite 1 Dokument(e)... → 🔄 round: Runde 1/3 → 🔄 tool: Web-Validierung → 🔄 agent: Moderator
6. Prüfe logs/<session>.jsonl → Sieh den exakten parsed content im ersten Trace-Eintrag.

###  Hardening & Audit-Hinweise

1. Network Isolation: --host 127.0.0.1 bindet Chainlit nur lokal. Für Remote-Zugriff nutze caddy/nginx + TLS, niemals 0.0.0.0 ohne Reverse-Proxy.

2. Process Sandbox: ProtectSystem=strict + ReadWritePaths verhindert, dass der Agent außerhalb von logs/ und memory/ schreibt.
Log-Integrität: logs/app.jsonl + logs/*.jsonl sind zeilenweise JSON. Jede Zeile ist atomar. Bei Absturz bleibt die letzte Zeile intakt.

3. Key-Rotation: Ändere Keys in .env, dann systemctl --user restart debate-agent@$(whoami). Kein Neustart des LLM-Servers nötig.

4. Disk-Usage Monitoring: du -sh logs/ + logrotate hält Storage unter Kontrolle. backup-Skript sichert vor Wartung.
